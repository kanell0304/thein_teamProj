# 모먹지 (Momeogji)

> 채팅방 안에서 모임 조건과 개인 선호를 모아 음식점 후보 추천 → 투표 → 최종 공지까지 한 번에 끝내는 서비스

> **주의**: 본 문서는 초안(draft)이며, Notion의 [`v3. 전체 흐름 정리`](https://app.notion.com/p/39ddb803401e809d8102eb06f64834cd) 기준으로 작성되었습니다. 기획 및 개발 진행 상황에 따라 지속적으로 수정됩니다.

## 📌 프로젝트 소개

`모먹지`는 채팅방 안에서 모임 장소를 정할 때 반복되는 번거로운 과정 — 참여자 파악, 각자의 조건(장소·예산·분위기) 취합, 음식점 후보 검색, 투표, 공지 — 을 채팅방을 벗어나지 않고 하나의 흐름으로 압축하는 서비스입니다.

주최자가 채팅방의 기능 아이콘 버튼으로 모임을 시작하면, 참여자들이 개인 조건을 입력하고, AI가 이를 종합해 실제 음식점 후보를 추천하며, 투표를 거쳐 최종 음식점과 약속 정보를 채팅방 상단에 공지합니다.

초기 MVP는 반복 빈도가 높고 의사결정 비용이 분명한 **직장 점심과 소규모 회식**을 주요 사용 사례로 삼습니다.

### 제안 배경

카카오톡 단체 대화방에서 약속을 정할 때 다음과 같은 문제가 반복됩니다.

- 실제 참여자가 명확하지 않다.
- 장소, 음식 종류, 예산, 분위기 등 각자의 조건이 다르다.
- 일부 사람만 의견을 내고 나머지 사람의 선호는 반영되기 어렵다.
- 누군가가 채팅방과 지도 앱을 오가며 검색과 정리를 전담해야 한다.
- 여러 장소 링크가 공유되어도 최종 결정까지 이어지지 않는 경우가 많다.
- 음식점이 정해진 뒤에도 일정 공유와 공지 과정이 남는다.

> **핵심 가치**: 채팅방 안에서 흩어진 조건과 개인 선호를 구조화하고, 실제 음식점 후보 추천부터 투표와 최종 공지까지 하나의 흐름으로 압축한다.

## 🔄 전체 흐름

1. **모임 시작** — 주최자가 채팅방 기능 아이콘 버튼으로 모임 생성 카드를 연다.
2. **공통 필수 설정** — 주최자가 대상 인원, 약속 날짜·시각, 만남 장소(기준 좌표 + 도보 5/10/15분 범위), 시간대, 목적, 투표 마감 시각을 입력한다.
3. **참여자 확정** — 주최자가 선택한 인원에게만 개별 필수 창이 전송되고, 각자 제외 음식·예산 등 개인 조건과 마이데이터 활용 동의 여부를 제출해야 참여자로 확정된다.
4. **AI 추천** — 확정된 참여자의 조건(+동의자의 목업 마이데이터 가공 결과)을 종합해, 장소 API로 조회한 실제 음식점 후보 3~5개와 그룹 단위 추천 이유를 생성한다.
5. **투표 및 재추천** — 참여자는 음식점 후보와 "재추천" 선택지에 중복 투표할 수 있다. 재추천이 단독 1등이면 기존 후보를 제외하고 새 회차를 연다.
6. **최종 공지** — 음식점이 확정되면 채팅방 상단에 음식점, 지도 정보, 약속 일시, 참여자 목록을 공지하고, 약속 시각까지 고정한다. 주최자의 수정 내역은 이력으로 남는다.

## 🧪 v3(MVP) 구현 범위

| 구분 | v3 적용 범위 |
|---|---|
| 채팅 화면 | 실제 카카오톡이 아닌 React 기반 목업 채팅방 |
| 대화 데이터 | 목업 채팅 데이터를 PostgreSQL에 저장 |
| 마이데이터 | 실제 마이데이터가 아닌 카드 결제 내역 형태의 목업 데이터. 동의한 경우에만 비동기로 가공 |
| 장소 정보 | 지도·장소 API에서 조회한 실제 후보 데이터 기준 |
| AI 역할 | 조건 해석, 후보 필터링·재정렬, 그룹 단위 추천 이유 생성 (음식점명·주소·가격 등 사실은 직접 생성하지 않음) |
| 투표 | 음식점 후보 + 재추천 선택지를 포함한 구조화된 투표 카드 |
| 최종 결과 | 채팅방 상단 공지, 지도 목적지, 약속 일시, 참여자 목록 |

**v3 범위에서 제외**: 실제 카카오톡 네이티브 연동, 실제 금융 마이데이터 연결, 예약 자동 완료, 카카오페이 결제·정산 자동화. 예약/정산은 음식점 결정 흐름이 검증된 이후 확장 기능으로 검토합니다.

## ⚠️ 리스크

- **개인정보 부담**: 마이데이터 활용은 참여자가 매 모임마다 명시적으로 동의/비동의를 선택해야 하며, 동의하지 않아도 모임 참여는 가능하도록 설계
- **AI 추천 정확도**: 부정확한 추천이 반복되면 기능 자체에 대한 신뢰뿐 아니라 AI/서비스 전반에 대한 신뢰 저하로 이어질 수 있음. 이에 따라 AI는 확인되지 않은 정보를 사실처럼 단정하지 않고 `미확인`으로 표시

## 🛠️ 기술 스택

| 영역 | 기술 |
|---|---|
| Backend | Java 21, Spring Boot 3.5.16 (Spring Web, Spring Data JPA, Validation, Security, WebSocket) |
| Database | PostgreSQL |
| Frontend | React 19.2.7, Vite |
| 빌드 도구 | Gradle 8 |
| AI 추천 | OpenAI API |
| 지도/위치 | 카카오맵 API |

REST API는 모임/참여자/투표/공지 등의 생성·조회·수정을 담당하고, WebSocket은 참여 상태 변화·추천 회차 시작·투표 마감·최종 공지 등의 실시간 알림을 담당합니다.

## 📂 프로젝트 구조

```
momeogji/

```
